# 智能药盒小程序设计文档

本文档基于微信官方《小程序设计指南》整理，并结合本项目的老人端、子女端、药品管理、血压记录等页面制定。后续新增页面、组件和视觉调整应优先遵循本文档。

官方参考：https://developers.weixin.qq.com/miniprogram/design/

## 1. 设计目标

智能药盒是一个面向老人和子女协作的健康管理小程序。设计目标不是“炫”，而是让老人能快速完成服药确认，让子女能稳定了解状态、补充药品信息、处理异常。

核心体验目标：

- 老人端：信息少、字大、按钮大、状态明确，关键操作一步完成。
- 子女端：信息密度更高，但仍需清晰分组，便于快速判断异常和进入详情。
- 全局：减少输入，优先使用拍照识别、选择器、预设时间、状态标签。
- 风险场景：漏服、血压异常、药品余量不足必须突出，并提供下一步操作。

## 2. 微信官方设计原则在本项目中的落地

### 2.1 友好礼貌

官方强调界面应重点突出、流程明确。本项目应做到：

- 每个页面只突出一个当前最重要任务。
- 老人端首页优先展示漏服提醒和下一次待服药。
- 鼓励文案使用温和、正向语气，例如“今天的药都吃完啦”“真棒”。
- 异常提醒避免责备语气，使用“超时未服”“待补药”等客观描述。

### 2.2 清晰明确

官方强调导航明确、反馈及时、异常可控。本项目应做到：

- 底部 Tab 只用于页面主分区切换，老人端为“今日用药 / 血压”，子女详情为“首页 / 药品 / 血压心率”。
- 页面内返回入口必须明确，文本建议为“返回”“取消”“重拍”等直接动作。
- 点击“我吃了”后必须在原按钮位置反馈“已记录”，避免用户怀疑是否成功。
- 拍照识别、保存血压、绑定邀请等流程完成后使用结果页或明确成功状态。
- 表单错误应原位提示，不应只用一闪而过的 toast。

### 2.3 便捷优雅

官方强调减少输入、避免误操作、充分考虑触控热区。本项目应做到：

- 药品添加优先拍照识别；无法识别时再由子女补充。
- 服药时间使用预设时间按钮，避免老人键盘输入。
- 主要按钮高度不低于 `88rpx`，老人端关键按钮建议 `104rpx` 以上。
- 底部 Tab、服药按钮、拍照按钮必须留足点击热区，避免靠近屏幕边缘。
- 不在同一区域放置多个相似主按钮，防止误触。

### 2.4 统一稳定

官方强调不同页面间控件和交互方式保持一致。本项目应做到：

- 统一使用暖橙色作为主操作色：`#E8924A`。
- 统一使用米白背景：页面背景 `#FAF6EE`，卡片背景 `#FDFBF7`。
- 统一底部 Tab 组件：`components/nav-item`。
- 统一趋势图组件：`components/line-trend`。
- 统一服药按钮组件：`components/take-med-button` 或卡片内同规格按钮。
- 新增页面必须复用现有 spacing、radius、color token，避免临时造一套样式。

## 3. 信息架构

### 3.1 角色入口

文件：`pages/index/index`

目标：让用户快速选择身份。

规则：

- 两个角色入口必须等权展示。
- “我为自己使用”进入老人端。
- “我帮家人管理”进入子女端。
- 不添加营销式内容，避免干扰选择。

### 3.2 老人端

页面：

- `pages/elderly/home/home`：今日用药、血压记录主页面。
- `pages/elderly/meds/meds`：我的药盒。
- `pages/elderly/photo-add/photo-add`：拍照添加药品。
- `pages/elderly/reminder/reminder`：全屏服药提醒。

优先级：

1. 漏服提醒。
2. 下一次待服药。
3. 子女鼓励。
4. 其他待服药。
5. 已服药记录。
6. 药盒管理入口。

老人端设计要求：

- 字号偏大，主信息建议 `36rpx` 以上。
- 卡片间距清晰，避免密集列表。
- 关键按钮使用高对比，文字直接使用“我吃了”“稍后提醒”。
- 血压趋势和记录使用简洁标签，不堆叠过多解释文字。

### 3.3 子女端

页面：

- `pages/child/overview/overview`：全部老人概览。
- `pages/child/elder-detail/elder-detail`：单个老人详情。
- `pages/child/bind-flow/bind-flow`：绑定老人。
- `pages/child/add-med/add-med`：添加药品。

子女端设计要求：

- 概览页突出异常：漏服、血压异常、余量不足。
- 详情页使用分区卡片：通知、今日用药、血压心率、药品管理。
- 药品日历用于复盘，不应抢占首页首屏。
- 添加药品表单必须清晰标注必填项，提交失败要说明具体原因。

## 4. 视觉规范

### 4.1 色彩

现有 token 位于 `styles/variables.wxss`。

主色：

- 主操作橙：`#E8924A`
- 深橙强调：`#D97D38`
- 浅橙底：`#F0A868`

背景：

- 页面背景：`#FAF6EE`
- 卡片背景：`#FDFBF7`
- 分割线/弱边框：`#F3EBDD`

文字：

- 主标题：`#292524`
- 正文：`#44403C`
- 次要文字：`#78716C`
- 辅助文字：`#A8A29E`

状态色：

- 成功：`#22C55E`
- 异常/漏服：`#F43F5E` 或 `#E11D48`
- 警告/补服：`#FACC15`

使用规则：

- 主色只用于主按钮、选中态、关键状态。
- 漏服和血压异常必须用玫红/红色区分，不用橙色表达风险。
- 卡片背景和页面背景要有轻微层级差，不使用大面积纯白。

### 4.2 字体

微信官方建议跟随系统字体。本项目全局使用：

```css
-apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, sans-serif
```

字号建议：

- 辅助信息：`22rpx`
- 正文：`26rpx` - `28rpx`
- 卡片标题：`32rpx` - `36rpx`
- 老人端关键时间：`52rpx` - `60rpx`
- 结果页标题：`44rpx`

规则：

- 老人端不使用小于 `24rpx` 的关键文字。
- 不使用负字距。
- 长药名允许换行，避免压缩到不可读。

### 4.3 间距和圆角

基础间距：

- 小间距：`8rpx`
- 常规间距：`16rpx` / `24rpx`
- 页面边距：`32rpx` - `40rpx`
- 卡片内边距：`28rpx` - `40rpx`

圆角：

- 小标签：`12rpx` - `16rpx`
- 普通卡片：`20rpx` - `28rpx`
- 主卡片/大按钮：`24rpx` - `28rpx`
- 胶囊标签：`9999rpx`

规则：

- 页面 section 不做嵌套卡片。
- 只有重复项目、表单容器、弹窗、卡片使用边框容器。
- 卡片圆角不超过 `28rpx`，除非是头像或胶囊标签。

## 5. 组件规范

### 5.1 底部 Tab

组件：`components/nav-item`

规则：

- Tab 数量不少于 2 个，不超过 5 个；本项目优先不超过 3 个。
- 每项必须平均分布，点击区域占满对应宽度。
- 老人端 Tab 需要更大尺寸，当前规格：
  - 底栏高度：`132rpx + safe-area`
  - 图标：`56rpx`
  - 文案：`26rpx`
- 当前选中态使用主橙色和浅橙图标底。
- 页面内容底部必须预留 `172rpx + safe-area`，避免被底栏遮挡。

### 5.2 主按钮

主按钮用于确定性操作，例如“我吃了”“完成”“保存记录”。

规则：

- 主按钮背景：`#E8924A`
- 主按钮文字：白色或深橙，取决于按钮背景。
- 老人端关键按钮宽度建议 `480rpx` 或满宽，不能过窄。
- 禁用态使用灰底和灰字。
- 点击后原位反馈，不跳动布局。

### 5.3 卡片

卡片用于承载独立任务或记录。

类型：

- 下一次服药卡：高优先级，橙色背景。
- 漏服卡：灰底或红色提示，强调“超时未服”。
- 普通药品卡：白/米白背景，展示时间、药名、剂量。
- 血压记录卡：使用左边色条表达异常状态。

规则：

- 卡片内最多一个主操作按钮。
- 状态标签放右侧或标题附近。
- 不在卡片里堆叠说明文字，必要说明放次级文字。

### 5.4 图标

图标来源：`utils/icons.js`。

规则：

- 图标风格统一为圆角线性，线宽 `1.8` - `2.5`。
- 底部 Tab 图标必须简洁，避免过细。
- 品牌/对象入口图标可以使用 CSS 绘制或专门 SVG，但不得使用粗糙临时图形。
- 图标与文字含义必须一致，不使用装饰性图标替代功能说明。

### 5.5 表单

规则：

- 必填项使用 `*` 标注。
- 输入框应有 label，不只依赖 placeholder。
- 时间、剂量单位、提醒频次优先选择器，不优先键盘输入。
- 错误提示要指出具体字段，例如“请填写药品名称”。

## 6. 交互和反馈规范

### 6.1 服药确认

流程：

1. 用户点击“我吃了”。
2. 按钮立即进入反馈态。
3. 展示“已记录”。
4. 药品状态更新为已服用或补服。

规则：

- 不用 toast 替代按钮原位反馈。
- 不在确认前增加二次弹窗，避免老人操作成本。
- 漏服后的“我吃了”记录为补服，但前端统一显示“已记录”。

### 6.2 拍照添加药品

流程：

1. 拍摄药盒。
2. 展示识别结果。
3. 选择服药时间。
4. 发送给子女补充剂量等信息。

规则：

- 每一步只出现一个主按钮。
- 拍照页使用深色背景和取景框。
- 识别结果页使用卡片承载照片和识别药名。
- 失败时提供“重拍”或“发给子女确认”。

### 6.3 血压记录

流程：

1. 拍照或手动输入。
2. 展示可修改识别结果。
3. 保存记录。
4. 成功页展示高压、低压、心率。

规则：

- 高压、低压、心率并列展示。
- 异常数值使用红色。
- 趋势图警戒线必须有标签。

### 6.4 异常状态

异常类型：

- 漏服。
- 血压或心率异常。
- 药品余量不足。
- 识别失败。
- 绑定失败。

规则：

- 异常要说明原因和下一步动作。
- 不让用户停留在无路可退的页面。
- 需要用户明确知晓的异常使用弹窗或结果页。
- 轻量成功反馈可使用原位反馈或短提示。

## 7. 适老化专项规范

本项目老人端必须优先考虑适老化。

要求：

- 主操作按钮不小于 `88rpx` 高。
- 关键文字不小于 `28rpx`。
- 时间、药名、剂量为分层展示，不放在一行里挤压。
- 不依赖颜色唯一传达状态，状态应有文字标签。
- 不使用复杂手势作为唯一操作方式。
- 滑动趋势图可作为增强操作，但必须保留按钮/列表查看方式。
- 页面首屏只保留核心任务，不添加营销内容。

## 8. 页面验收清单

新增或修改页面时逐项检查：

- 当前页面最重要任务是否在首屏可见？
- 是否有明确返回路径？
- 是否有超过一个主按钮同时出现？
- 点击后是否有及时反馈？
- 是否存在按钮过小、靠边、过密的问题？
- 底部 Tab 是否平均铺满，是否预留 safe-area？
- 长药名、长老人姓名是否会挤压布局？
- 异常状态是否有明确原因和下一步？
- 老人端关键文字是否足够大？
- 页面是否复用了现有颜色、间距、圆角和组件？

## 9. 当前项目优先优化建议

短期：

- 稳定老人端首页顶部布局，避免药盒入口与微信胶囊按钮冲突。
- 统一底部 Tab 在老人端和子女端的视觉尺寸。
- 将“我吃了”按钮宽度、字号、高度作为组件 token 固化。
- 调整所有药品长名称的换行和截断策略。

中期：

- 把主按钮、状态标签、卡片、页面头部抽成公共样式。
- 为血压异常、漏服、药品余量不足建立统一状态组件。
- 建立空状态、错误状态、加载状态组件。

长期：

- 引入真实设备预览清单，覆盖 iPhone SE、iPhone 12/13、安卓常见机型。
- 增加适老化模式开关，支持更大字号和更强对比度。
- 增加视觉回归截图检查，避免底部 Tab、顶部胶囊、安全区再次回退。
